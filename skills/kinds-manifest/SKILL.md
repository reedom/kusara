# kusara kinds manifest

The kinds manifest declares every valid `kind:` value. It lives at `${KUSARA_DOC_ROOT}/kinds.md` (default `docs/kinds.md`). When asked about kinds, **read the project's manifest first** ‚Äî this skill only describes the format and shipped defaults of the kusara repo itself.

## How to look up the truth

Always read the actual manifest before answering:

```sh
cat "${KUSARA_DOC_ROOT:-docs}/kinds.md"
```

The YAML block under `## Manifest` is the source of truth.

## Entry shape

```yaml
- name: <kind>             # kind identifier; goes into `kind:` and as id prefix
  path_globs: ["..."]      # repo-relative globs the scanner walks
  declared_via: <provides|generated>   # optional; informational
  id_pattern: "<pattern>"  # informational; e.g. "fr:{slug}"
  singleton: true          # optional; exactly one doc of this kind
  index:                   # optional; enable `kusara index` output
    output: <path>         # repo-relative path for the generated INDEX
    group_by: spec         # optional; split INDEX by `spec:` value
```

## Field semantics

- **`name`** ‚Äî the string used as `kind:` in frontmatter and as id prefix (e.g. `fr` ‚Üí `id: fr:01-foo`).
- **`path_globs`** ‚Äî repo-relative literal globs. Not prefixed by `KUSARA_DOC_ROOT`. Omit when the kind is declared only via `provides:`.
- **`declared_via: provides`** ‚Äî kind has no files of its own; ids appear inside another doc's `provides:` list. Typical: `req`.
- **`declared_via: generated`** ‚Äî kind is written only by `kusara index`. Reserved: `index`.
- **`id_pattern`** ‚Äî informational only. The validator does not enforce it; it enforces uniqueness alone.
- **`singleton`** ‚Äî informational. Implies exactly one file of the kind.
- **`index.output`** ‚Äî when set, `kusara index` writes a generated INDEX file there. The file carries `kind: index`, `generated: true`, `indexes_kind: <name>`.
- **`index.group_by: spec`** ‚Äî INDEX body splits into one section per `spec:` value.

## Reserved kind: `index`

Files generated by `kusara index` carry `kind: index` and are never hand-edited. Their frontmatter has `id: index:<name>`, `kind: index`, `generated: true`, and `indexes_kind: <name>`.

## Choosing a kind for a new doc

1. List the kinds in the project manifest.
2. Match the file's path against `path_globs` of each kind.
3. If exactly one matches ‚Üí that's the kind.
4. If none match ‚Üí the file path needs to move, OR a new kind needs to be added to the manifest.
5. If multiple match ‚Üí manifest bug; flag it.

## Adding a new kind

1. Append an entry to the YAML block under `## Manifest`.
2. Set `index.output` if you want `kusara index` to generate an INDEX.
3. Run `kusara validate`.

Removing a kind is destructive: any frontmatter using it will fail validation.

## Defaults shipped with the kusara repo

The kusara repo's own `docs/kinds.md` defines: `roadmap` (singleton), `spec`, `fr`, `reference`, `req` (declared_via provides), `index` (declared_via generated). Project manifests vary ‚Äî always read the project's own file before answering.
