# Doc Kinds Manifest

Kind definitions consumed by `kssni`. The CLI loads `${KSSNI_DOC_ROOT}/kinds.md` at startup (default `KSSNI_DOC_ROOT=docs`); the YAML block below is the source of truth.

Schema semantics (`implements` / `depends_on` / `related` / `provides` / `modules`): [`refs.md`](refs.md).

## Adding a new kind

1. Add an entry to the YAML block.
2. Add `index.output` if a generated INDEX is wanted.
3. Run `kssni validate`.

## Manifest

```yaml
# Each entry is a kind definition.
#   name           — kind identifier used as the `kind:` field in front matter
#                    and as the prefix in `id:` (e.g., kind=fr → id=fr:01-...).
#   path_globs     — repo-relative globs the scanner walks. Literal paths;
#                    not prefixed by KSSNI_DOC_ROOT. Omit for kinds declared
#                    only via `provides:` of another doc.
#   declared_via   — "provides" if this kind never has its own file
#                    (the IDs are listed in the `provides:` of another doc),
#                    or "generated" for kinds written only by `kssni index`.
#                    Informational; only its presence/absence is enforced.
#   id_pattern     — informational; what an ID for this kind looks like.
#                    Not enforced by the validator.
#   singleton      — true if there is exactly one doc of this kind
#                    (e.g., roadmap). Informational.
#   index          — when set, `kssni index` writes a generated
#                    INDEX for this kind to index.output. The generated
#                    file carries frontmatter (kind=index, generated=true,
#                    id=index:<name>, indexes_kind=<name>).
#       output     — repo-relative path to write the INDEX to.
#       group_by   — optional. Recognized values: `spec`. When set to
#                    `spec`, the INDEX body is split into one section
#                    per `spec:` value.
kinds:
  - name: roadmap
    path_globs: ["docs/roadmap.md"]
    singleton: true
    id_pattern: "roadmap"

  - name: spec
    path_globs: ["docs/specs/*.md"]
    id_pattern: "spec:{slug}"
    index:
      output: docs/specs/index.md

  - name: fr
    # Functional requirement pages: docs/fr/01-foo.md, 02-bar.md, ...
    path_globs: ["docs/fr/[0-9]*.md"]
    id_pattern: "fr:{slug}"
    index:
      output: docs/fr/index.md

  - name: reference
    # Glob `[a-z]*.md` matches lowercase filenames; `README.md` and other
    # uppercase filenames are not matched. The generated `index.md` is
    # matched and carries its own `kind: index` frontmatter.
    path_globs: ["docs/reference/[a-z]*.md"]
    id_pattern: "ref:{slug}"
    index:
      output: docs/reference/index.md

  - name: req
    declared_via: provides
    id_pattern: "req:{spec}:{n}"

  - name: index
    declared_via: generated
    id_pattern: "index:{kind}"
```

## Reserved kind: `index`

Files generated by `kssni index` carry `kind: index` and are never hand-edited. Their frontmatter contains `id: index:<name>`, `kind: index`, `generated: true`, and `indexes_kind: <name>` (per-kind indexes); the body is the table of indexed docs.

## Out of scope

- No glob expansion beyond standard shell globbing.
- `id_pattern` is documentation, not enforcement; uniqueness is the only ID check.
- Kinds with no `path_globs` and no `declared_via` fail validation.
