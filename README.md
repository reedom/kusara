# kusara

Markdown cross-reference graph CLI: validate dangling refs, generate indexes, trace impact between specs, docs, and code.

## What it does

You/AI hand-edit `refs:` YAML frontmatter on Markdown files. `kusara` reads it as a graph and answers:

- **Did I break a link?** (`kusara validate` flags dangling refs, duplicate IDs, unknown kinds, missing source paths.)
- **What docs cover this code file?** (`kusara touched src/auth/session.rs`)
- **If I change spec X, what else needs updating?** (`kusara impact spec:auth`)
- **What does doc Y depend on?** (`kusara deps fr:login`)
- **Give me a per-kind index page.** (`kusara index`)

Hand-edited frontmatter, machine-checked graph. No database, no daemon â€” just a CLI over your Markdown.

## Install

Requires Rust toolchain (install via [rustup](https://rustup.rs)).

```sh
cargo install kusara
```

Or build from a clone:

```sh
git clone https://github.com/reedom/kusara && cd kusara
cargo install --path .
```

Verify:

```sh
kusara --version
```

## Quick start

### 1. Tell kusara what kinds of docs you have

Create `docs/kinds.md` in your repo (the path is `${KUSARA_DOC_ROOT}/kinds.md`; default `KUSARA_DOC_ROOT=docs`). Minimal example:

````markdown
# Doc Kinds Manifest

```yaml
kinds:
  - name: reference
    path_globs: ["docs/*.md"]
    id_pattern: "ref:{slug}"
```
````

Each `kind` is a doc category you define (reference, spec, decision, fr, ...). `path_globs` lists files of that kind. See [`docs/kinds.md`](docs/kinds.md) in this repo for a fuller example with per-kind index generation.

### 2. Add `refs:` frontmatter to a Markdown file

```markdown
---
refs:
  id: ref:auth-overview
  kind: reference
  title: "Auth subsystem overview"
  modules:
    - src/auth/
---

# Auth Overview
...
```

Required: `id`, `kind`. Optional: `title`, `spec`, `provides`, `implements`, `depends_on`, `related`, `modules`. Schema reference: [`docs/refs.md`](docs/refs.md).

### 3. Run

```sh
kusara validate                            # check the graph
kusara list                                # every known ID
kusara show ref:auth-overview              # one doc + its edges
kusara impact ref:auth-overview            # who depends on this?
kusara touched src/auth/session.rs         # which docs cover this file?
kusara index map                           # write map.md + ai/graph.json + ai/modules.md
kusara index                               # write per-kind INDEX files
```

### Sample output

```
$ kusara validate
OK (12 docs)

$ kusara touched src/auth/session.rs
Files:
  src/auth/session.rs

Docs of record (modules: covers these files):
  ref:auth-overview                        Auth subsystem overview

Transitively affected:
  fr:login                                 Login flow
  spec:auth-v2                             Auth v2 spec
```

## Subcommands

```sh
kusara validate
kusara impact <id> [<id>...] [--depth <N>] [--include-related]
kusara deps   <id> [<id>...] [--depth <N>] [--include-related]
kusara show   <id>
kusara touched <file> [<file>...] [--no-closure]
kusara list
kusara index map
kusara index
```

`--root <DIR>` (global): override the repo root (default: cwd).

## Configuration

`KUSARA_DOC_ROOT` (default `docs`) points at the directory containing `kinds.md` and where `map.md` / `ai/graph.json` are written.

## Troubleshooting

- **`read kind manifest at docs/kinds.md`** â€” file missing. Create `docs/kinds.md` or set `KUSARA_DOC_ROOT` to the directory that contains it.
- **`unknown kind 'X'`** â€” the doc's `kind:` is not declared in `kinds.md`. Add the kind to the manifest.
- **`dangling reference 'Y'`** â€” `Y` is not the `id:` (or `provides:`) of any doc in the graph. Fix the typo or add the missing doc.
- **`duplicate id 'Z'`** â€” two docs claim the same id. IDs must be unique across the repo, including `provides:` aliases.
- **`map.md` / `index.md` shows up as modified** â€” these files are generated by `kusara index`. Don't hand-edit them; rerun `kusara index` after content changes.
